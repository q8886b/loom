# 第二大脑通用版方案（活文档）

> 本文档记录当前讨论中已经确认的通用版 Loom 方案。  
> 更新规则：后续每次增量修改前，先在对话中说明拟更新内容，经确认后再写入。

---

## 1. 系统定位

Loom 不是医学专用知识库，也不是更复杂的 RAG 管线。它是一个通用知识卡片网络：把任何领域的原始材料，转成可独立理解、可互相连接、可持续生长的知识卡片。

适用领域包括但不限于：

- 医学
- 软件工程
- 产品与业务
- 组织管理
- 法律
- 哲学与认知
- 科研与论文
- 个人经验与复盘

医学只是其中一个 namespace，不是系统底层假设。

---

## 2. 卡片本体

卡片记录的是已经经过选择、裁剪、整理、改写或综合后的知识单元，不是原始材料本身。

卡片的最小结构：

```json
{
  "id": "med:1a1a1",
  "title": "黏液瘤栓塞机制",
  "type": "机制",
  "content": "黏液瘤的胶冻状质地和带蒂活动性，使其在血流冲击下容易碎裂或形成表面血栓，脱落后进入体循环造成脑、肾、冠脉或肢体栓塞。见 med:1a1a（病理结构解释了为什么容易碎裂），见 gen:2a（不稳定结构在传播路径中造成远端后果）。",
  "links": ["med:1a1a", "gen:2a"],
  "source": "Harrison's Principles of Internal Medicine, 22e, Chapter 282: Atrial Myxoma and Other Cardiac Tumors, pp.2127-2129。由关于左房黏液瘤病理结构、带蒂活动性和栓塞并发症的段落整理。"
}
```

字段含义：

| 字段        | 含义                             |
| --------- | ------------------------------ |
| `id`      | namespace + 卢曼 ID，表示知识空间和主生长路径 |
| `title`   | 人类浏览用短标题                       |
| `type`    | 内容类型 / 导航路由字段，不塞进 title        |
| `content` | 卡片主体；保存消化后的知识内容，也自然表达与其他卡的关系   |
| `links`   | 双向图邻接边，只负责快速跳转                 |
| `source`  | 描述性来源文本，能找到原始内容来自哪里            |

---

## 3. content 的性质

`content` 不强制必须重写。它可以是以下几种形态：

1. **原文裁剪**：原文表达已经足够精确，直接裁剪保存。
2. **裁剪整理**：从原文多个片段中裁剪、拼接、轻度整理。
3. **消化改写**：用自己的话压缩、解释、重组。
4. **综合抽象**：从多个来源或多张卡中抽象出新的认识。

关键标准不是“是否原创表达”，而是：

- 独立可读
- 粒度合适
- 可被其他卡引用
- 可通过 source 追溯
- 可通过 links 连接

---

## 4. links 的规则

`links` 是图结构，不承载关系解释。

- `links` 只保存相连卡片 ID。
- 建 link 时双向都建。
- 关系含义写在 `content` 中的自然语言里，通常采用：`见 <card_id>（为什么见它）`。

例如：

```text
content: "……见 med:1a1a（病理结构解释了为什么容易碎裂），见 gen:2a（不稳定结构在传播路径中造成远端后果）。"
links: ["med:1a1a", "gen:2a"]
```

建图时：

```text
med:1a1a1.links 包含 med:1a1a
med:1a1a.links 也包含 med:1a1a1
```

也就是说，`links` 是无向图邻接表；`content` 负责说明为什么相连。

---

## 5. source 的规则

`source` 是描述性来源文本，不做复杂嵌套结构。

它要回答：这张卡来自哪里？如果要核查，应该去哪找原始内容？

示例：

```text
Harrison's Principles of Internal Medicine, 22e, Chapter 282: Atrial Myxoma and Other Cardiac Tumors, pp.2127-2129。由关于左房黏液瘤病理结构、带蒂活动性和栓塞并发症的段落整理。
```

```text
OpenMed 后台任务排障会话，2026-06，关于长任务输出缓冲和后台进程未及时停止的讨论整理。
```

```text
从 med:1a1a1、swe:2b1 等卡片中抽象整理。
```

---

## 6. type 的规则

`type` 单独存字段，用于导航和检索路由，不写进 title。

`type` 表示卡片在知识网络中的主要功能，不表示严格线性层级。第一版要求一张卡只有一个主 `type`。后续如有需要，再讨论是否引入 tags。

### type 自检规则（建卡时强制执行）

每张卡创建前，必须用以下决策树确定 type，不得凭直觉分配。决策树既可用于"已有内容归为哪类"（判断 type），也可用于"扫描材料时应识别什么建卡"（识别信号）：

| type | 决策树问题（type 判断） | 材料识别信号（建卡触发） |
|---|---|---|
| **判断** | 这张卡是否在回答"基于证据应该怎么选/怎么做？" | 材料给出了基于证据的选择、评价、决策理由、证据权衡 |
| **机制** | 是否在解释"X 导致 Y，因果链是什么？" | 材料描述了"如果 X 则 Y"的因果、过程、触发条件、反馈循环 |
| **结构** | 是否在描述"它由什么组成？空间/层次结构？" | 材料列举了体系、组成、层次、顺序、拓扑、流程 |
| **概念** | 是否在回答"这到底是什么？定义、边界、分类？" | 材料定义了一个术语、概念、概念边界、分类归属 |
| **比较** | 是否在对比"A vs B 的异同"？ | 材料对比/区分/列出了两个或多个对象的异同、例外 |
| **案例** | 是否在记录一个具体实例/事件？ | 材料记录了具体实例、事件、判例、事故、用户反馈 |
| **反思** | 是否在质疑/修正已有判断？ | 材料指出了适用边界、反例、误用、冲突、修正方向 |
| **模式** | 是否从多个案例/判断中抽象共同模式？ | 材料从多个案例、判断、领域中抽象出可迁移的稳定结构 |

第一版 type：

- **概念**：回答“这是什么”。包括定义、概念边界、分类归属和分型。
- **结构**：回答“它由什么组成 / 在哪里 / 怎么排列”。包括组成、空间、层级、顺序、拓扑和流程结构。
- **机制**：回答“它如何发生 / 为什么变化”。包括因果链、过程、反馈循环、条件触发和系统动力学。
- **案例**：回答“具体发生过什么”。包括病例、事故、实验、判例、历史事件、用户反馈和项目复盘。
- **比较**：回答“A 和 B 有什么同异”。包括鉴别、方案比较、类案区分、概念辨析、类比和差异。
- **判断**：回答“基于材料我怎么看 / 怎么选”。包括结论、评价、证据权重、取舍和决策理由。
- **反思**：回答“这个判断或模式哪里需要检查”。基于某个判断或模式，说明适用条件、例外、反例、冲突、误用或修正方向。
- **模式**：回答“多个案例或判断背后的共同结构是什么”。从多个案例、判断或领域中抽象出可迁移的稳定模式；成熟后可在 title/content 中称为原理。

认知层级说明：

1. **一阶内容卡**：概念 / 结构 / 机制 / 案例 / 比较。
2. **二阶结论卡**：判断。案例和判断不是同一认知层级；案例通常作为判断的材料。
3. **元层卡**：
   - 模式：从多个案例或判断中向上抽象共同结构。
   - 反思：对判断或模式回头校正，检查适用条件、反例、冲突、误用和修正方向。

第一版不单列“定义”“分类”“证据”“决策”“因果”“类比”“原理”：

- 定义、分类归入“概念”。
- 证据评价、权衡取舍和行动选择归入“判断”。
- 因果扩展为“机制”，以覆盖过程、反馈和条件触发。
- 类比扩展为“比较”，以覆盖相似、差异、鉴别、方案比较和概念辨析。
- 原理收敛为“模式”，避免一开始就把抽象结果视为已验证定律。

### 6.1 type 体系的理论来源

这套 `type` 不是领域分类，也不是封闭的终极范畴表，而是 Loom 中卡片的功能分类：它描述一张卡在知识网络里主要承担什么认知作用。

它的目标是“工作完备”而不是哲学上绝对完备：

1. 大多数卡片都能找到一个主要功能。
2. 类型能指导 AI 检索、导航、生成和反思。
3. 类型数量不能太多，避免建卡时过度纠结。
4. 类型体系允许未来随着使用继续演化。

理论来源：

- **康德**：
  - 规定性判断力对应“概念”：把特殊归入已有普遍，回答“这是什么”。
  - 时空直观与知性组织经验，对应“结构”和“机制”。
  - 反思性判断力对应“反思”和“模式”：面对特殊情况时，不只是套用既有规则，而是为特殊寻找新的普遍。
  - 实践理性对应“判断”中的行动选择：在目标和约束下为什么这样做。
- **叔本华四重根**：
  - 生成根据：为什么会发生 → “机制”。
  - 存在根据：为什么在这里 / 为什么是这种结构 → “结构”。
  - 认识根据：为什么相信这个判断 → “判断”中的证据判断。
  - 行动根据：为什么选择这个行动 → “判断”中的决策判断。
- **皮亚杰与发生认识论**：
  - 知识不是被动接收，而是在同化和顺应中建构。
  - “案例”提供经验材料；“判断”是对材料的组织；“反思”对应顺应和框架修正；“模式”对应反省抽象。
- **认知科学**：
  - 原型理论与图式形成支撑“概念”。
  - 结构映射理论支撑“比较”。
  - case-based reasoning / 情景记忆支撑“案例”。
  - 元认知、双环学习和范式转换支撑“反思”。
- **认知神经科学**：
  - 不同认知任务依赖不同但相互连接的神经网络；从具体规则加工到抽象整合是连续梯度，不是硬断裂。
  - 因此 `type` 用于路由和组织，而不是把知识锁死在互斥盒子里。

关键辨析：

- **模式 vs 原理**：模式是从多个案例或判断中抽象出的可迁移共同结构；原理是反复验证、边界清楚、已经稳定下来的模式。第一版 type 使用“模式”，成熟后可在 title/content 中称为原理。
- **认识根据 vs 行动根据**：认识根据回答“为什么相信这个判断”，行动根据回答“为什么选择这个行动”。第一版都归入“判断”，但在 content 中要写清楚具体是哪一种根据。
- **判断、反思、模式的层次**：判断基于材料形成结论；反思检查某个判断或模式的适用条件、反例、冲突和误用；模式从多个案例或判断中抽象共同结构。


### 6.2 消化密度门禁（学科通用）

> 以下规则适用于所有 namespace，不是某一领域的特殊要求。

#### Loom 的定位：思考伙伴（不是记忆系统）

Loom 是**有记忆的思考伙伴**。记忆是底座能力，**思考才是核心价值**。

它做四件事：
- **回忆**——带着已有卡片网络调出相关材料
- **联想**——从具体现象识别跨域模式（gen 模式卡）
- **比较**——跨域、跨概念对照（比较卡）
- **质疑**——对已有判断提出反例、边界、误用（反思卡）

**记忆系统（RAG/数据库）** 和 **思考伙伴** 的本质区别：
- 记忆系统：用户问 X，调出含 X 的内容，**到此为止**
- 思考伙伴：用户问 X，调出 X + 联想相关概念 + 跨域类似机制 + 反例边界 + 可行动判断，**带着思考**

由此推出消化的硬要求：
1. **覆盖每个独立认知单元**——不是为了完整存档，是为了未来思考时素材不缺失
2. **建立有意义的连接**——让联想可发生
3. **形成可调用的判断卡和反思卡**——让质疑和决策可发生

**消化 ≠ 完整存档 ≠ 读书笔记 ≠ 索引。消化 = 把材料里每一个能独立思考的认知单元都提取出来，建成可被未来思考调用的素材。**

#### 密度由材料的认知结构决定，不由字符数决定

一张卡 = 一个能独立回答明确问题的认知单元。不要按"每 X 字符建 Y 张卡"机械切分。识别材料中的认知单元的方法见上面的 **type 自检规则表**——它既给出了"判断 type"的决策树，也给出了"扫描材料时应识别什么建卡"的识别信号。

#### 量化密度诊断（工程化检查工具）

卡的数量不是目的，**覆盖度**才是目的。但工程上需要可量化诊断：

**完整版材料（>50 页）的密度参考**：每页约 1-2 张卡，即页/卡 < 1.5。

| 材料类型 | 页数 | 合理卡数 |
|---|---|---|
| 大全型（每种形态/概念独立成章）| 1000 页 | 500-2000 张 |
| 密集型（每章核心概念 + 多变体）| 300 页 | 200-500 张 |
| 案例型（大量故事 + 少量概念）| 300 页 | 100-300 张 |
| 经典型（每章节多独立原则）| 600 页 | 300-600 张 |
| 预览版/节选 | 5-30 页 | 视密度，30 页通常 15-40 张 |

**页/卡 > 3 几乎肯定漏了独立单元，是失败消化。** 这是诊断信号，不是机械指标。真正的判断标准还是"未来用户能不能调出对应卡"。

#### 覆盖要求

扫描材料时识别出的每个独立认知单元都应有对应的卡，除非已有旧卡覆盖同一单元。

#### 内容长度由"讲清楚这个单元"决定，不预设字数

- 简单概念可能 30 字就够
- 复杂机制可能 800 字才讲清楚
- 关键是：能独立读懂，不依赖原文上下文

#### 禁止的反模式（学科通用描述）

| 反模式 | 通用描述 | 为什么禁止 |
|---|---|---|
| **稀疏型（最常犯）** | 把整本材料压成几张代表性卡，页/卡 > 3 | 等于漏了 90% 独立单元；用户提问时调不出，loom 失效 |
| **总结型** | 1 张卡涵盖多个不相关的独立认知单元 | 卡片无法独立回答单一问题，检索时噪音大 |
| **目录型** | 只列名称/标题，不解释内容 | 无信息增值，等于没消化 |
| **抄录型** | 照抄原文不做消化改写 | 未真正理解，无法被其他卡有意义地引用 |
| **碎片型** | 把一个完整单元切成多张卡 | 破坏独立性，增加检索负担 |

判断"是否多个不相关单元"看的是材料结构本身——
- 一段讲同一概念的多角度展开 = 1 张卡
- 一段讲多个独立规则的列举 = 多张卡

#### links 的密度

数量本身不是质量问题。真正的硬约束是：

- **每条 link 都必须在 content 中有对应的"见 X（为什么见它）"**——无意义 link 是反模式
- 某些中心概念卡可能合理地连到很多相关卡（10+ 也正常）
- 某些边缘概念卡可能只有 1 条 link 甚至 0 条（独立存在）
- 关键看 links 是否都有 content 解释，而不是数字

#### 建卡前自检 checklist

- [ ] 这张卡是否只回答一个明确的问题？
- [ ] 不看原文能否独立读懂？
- [ ] 我是否在消化改写，而不是抄录？
- [ ] 我有没有用 links 表达关系，而不是塞进 content？
- [ ] 同一材料中是否有其他独立单元被我漏掉了？

#### 消化后自检 checklist

- [ ] **密度是否达标？** 完整版书页/卡 < 1.5；> 3 几乎肯定漏了
- [ ] **抽样验证调出能力**：随机抽 3-5 个具体概念/案例/判断，用 search 查，能调出对应卡吗？调不出 = 漏了
- [ ] 有没有重复？
- [ ] 有没有冲突？
- [ ] 有没有总结型 / 稀疏型？

#### 与 SKILL.md 的关系

`skills/second-brain/SKILL.md` 是认知循环的操作指南；本节是密度与粒度的硬门禁。两者冲突时，密度门禁优先。


## 7. ID 与 namespace

完整 ID 采用：

```text
<namespace>:<luhmann_id>
```

例如：

```text
med:1a1a1
swe:2b1
gen:2a
```

### namespace

namespace 用于大领域分割，避免不同学科的卢曼路径互相污染。

初步 namespace：

| namespace | 含义 |
|---|---|
| `med` | 医学 |
| `swe` | 软件工程 |
| `prd` | 产品 / 业务 |
| `mgt` | 管理 / 组织 |
| `phi` | 哲学 / 认知 |
| `law` | 法律 |
| `gen` | 通用原理 / 跨领域模式 |

### 卢曼 ID

namespace 内部使用卢曼式生长编号。

规则：

1. 根卡：`1`, `2`, `3` ...
2. 子卡编号根据父 ID 末尾决定：
   - 父 ID 末尾是数字 → 子卡用 `a`, `b`, `c` ...
   - 父 ID 末尾是字母 → 子卡用 `1`, `2`, `3` ...
3. 每增加一位表示深入一层。
4. `parent(id)` = 去掉最后一位。
5. `prefix_chain(id)` = 从左到右逐位累积。
6. 同父卡下按自然顺序递增。
7. ID 只表达主生长路径，不表达所有关系。
8. 横向关系由 `content` 中的 `见 X（原因）` 和双向 `links` 表达。

示例：

```text
med:1       心脏肿瘤
med:1a      原发性心脏肿瘤
med:1a1     黏液瘤
```

## 8. gen 公共原理空间

`gen` 是跨领域公共原理 / 通用模式空间。

领域卡负责具体知识，`gen` 卡负责从多个领域或多个案例中抽象出的模式。

示例：

```json
{
  "id": "gen:2a",
  "title": "不稳定结构在传播路径中造成远端后果",
  "type": "模式",
  "content": "当一个系统中存在不稳定结构，并且它位于某种流动、传播或依赖路径上时，它的破裂、脱落、失控或错误输出会沿路径传递到远端位置。医学中的肿瘤碎片栓塞、软件中的失控异步任务、组织中的错误激励扩散都体现了这个模式。这个原理不替代具体领域知识，它只是提示分析时要追问三个问题：不稳定结构是什么，传播路径是什么，下游承受点在哪里。见 med:1a1a1（黏液瘤碎裂导致体循环栓塞），见 swe:2b1（失控任务导致下游资源挤占）。",
  "links": ["med:1a1a1", "swe:2b1"],
  "source": "从 med:1a1a1、swe:2b1 等卡片中抽象整理。"
}
```

## 9.5 检索与使用方式

### 核心原则：专家式回忆，不是搜索引擎

Loom 的使用不是"搜一个词→看结果→结束"。它模拟的是有经验的专家遇到问题时的回忆和联想过程：

1. 想到一个方向
2. 顺着它展开
3. 展开的内容激发新方向
4. 再展开
5. 反复，直到足够

### 三种检索能力（互相激发）

1. **结构性回忆**：知道知识骨架，知道某个主题下有什么。
   - 对应工具：`browse`、`children`、`siblings`
   - 相当于专家"我知道这块的结构是什么"

2. **语义联想**：从描述中感觉到相关性。
   - 对应工具：`search`（向量 + 全文混合）
   - 相当于专家"这让我想起了..."

3. **模式/原理激活**：从具体现象中直觉性地识别出已有模式。
   - 对应工具：`search --ns=gen`、`browse gen`
   - 相当于专家"这看起来像是某个已知模式的又一个实例"

这三种不是三个独立步骤，而是在思考过程中自然交替、互相激发。

### 多轮回忆循环

```text
想到一个方向
  → 搜索/浏览
  → 找到相关卡
  → 看它的 neighbors
  → 从邻居中发现新方向
  → 再搜索/浏览
  → ...
  → 直到足够
```

不是一次搜索就停。每次搜到的东西都可能激发新的搜索方向。

### 认知循环精要

认知循环不是线性管线，是螺旋。每一步都产生新问题，每个新问题又需要新的搜索。

核心规则：
- **回忆阶段至少 3 轮搜索螺旋**（宽泛定位 → 沿链接深入 → 交叉验证），不是搜一次就停
- **每轮必须比上一轮更深**——走 neighbors、用更具体的关键词、发现新方向
- **产出后必须回头看**——重新搜索确认无重复/冲突/遗漏，回到材料确认无重要遗漏
- **至少用 2 种检索方式**（结构性回忆 / 语义联想 / 原理层扫描）

### 三个使用场景

| 场景 | 触发 | 输出 |
|---|---|---|
| digest | 给一段新材料 | 新卡 / 更新卡 / 反思卡 / 模式卡 |
| think | 问一个问题 / 讨论一个话题 | 答案（引用卡片）+ 可能的新卡 |
| review | 描述一次经历 / 错误 / 发现 | 案例卡 + 判断卡 + 可能的反思/模式 |

三个场景共享同一套底层能力（搜索、浏览、建卡、建link），由一个统一的 Agent 处理。

### 浏览页面（给人用）

人直接浏览卡片网络不需要经过 Agent，走独立前端页面：

- 卡片树视图（按 namespace + 卢曼前缀展开）
- 图视图（links 网络可视化）
- 搜索
- 点击卡片看完整内容
- 沿 links 跳转
- 看来源

这个页面调用的后端接口和 Agent 使用的工具是同一套（tools.py / store.py）。

## 10. 实现方式

### 10.1 项目位置

Loom 是通用领域实现，不绑定 OpenMed 医学项目：

```text
~/loom/                 ← 独立通用项目（可 clone 到任意位置）
~/OpenMed/backend/loom  ← 软链指向上面（如果某个项目想复用）
```

### 10.2 存储

主存储使用 SQLite 单文件，不用散落 JSON：

```text
loom/data/brain.db
```

表结构：

```sql
-- 卡片主表
CREATE TABLE cards (
    id TEXT PRIMARY KEY,          -- 'med:1a1a1'
    namespace TEXT,               -- 'med'
    luhmann_id TEXT,              -- '1a1a1'
    parent_id TEXT,               -- 'med:1a1a'
    title TEXT,
    type TEXT,                    -- '机制'
    content TEXT,
    source TEXT,
    created_at TEXT,
    updated_at TEXT
);

-- 双向 links
CREATE TABLE links (
    card_a TEXT,
    card_b TEXT,
    PRIMARY KEY (card_a, card_b)
);

-- 向量索引（sqlite-vec 扩展）
CREATE VIRTUAL TABLE card_vectors USING vec0(
    card_id TEXT,
    embedding FLOAT[2048]
);

-- 全文搜索
CREATE VIRTUAL TABLE cards_fts USING fts5(
    id, title, content, type, namespace
);
```

一个文件内包含：卡片存储、图结构、向量搜索、全文搜索。

### 10.3 Agent

承载方式：Claude Code 自定义 Agent（`.claude/agents/second-brain-digester.md`）。

Agent 不是无状态 prompt，而是带工具的智能体：

- 它带着已有卡片体系去阅读新材料
- 阅读过程中主动回忆（搜索旧卡）、联想（沿 links 跳转）、比较、质疑
- 最终把新材料消化进网络

Agent 启动时读取本文档（`docs/design/000-system-overview.md`）确保理解系统设计的前因后果、目的和精要。

### 10.4 工具脚本

轻量 Python 工具，供 Agent 通过 bash 调用：

```text
loom/tools.py
```

核心接口：

- `search_cards(query, namespace?, type?, top_k)` — 向量 + 全文搜索
- `get_card(card_id)` — 读取一张卡
- `get_neighbors(card_id, depth=1)` — 获取 links 邻居
- `browse_prefix(namespace, prefix)` — 浏览某卢曼前缀下所有卡
- `create_card(id, title, type, content, links, source)` — 创建卡 + 双向建 links
- `update_card(card_id, field, value)` — 更新卡片
- `add_link(card_a, card_b)` — 双向建 link

### 10.5 Agent 消化流程

```text
Pass 0: 理解材料 — 领域、namespace、主题、与已有网络可能的关系
Pass 1: 回忆已有卡片 — 搜索相关旧卡，浏览同 namespace 邻近卡
Pass 2: 识别知识单元 — 哪些内容值得长期保存，建议 type
Pass 3: 判断动作 — 新建 / 补充旧卡 / 拆分 / 建 link / 抽象模式
Pass 4: 写卡 — content 独立可读，见 X（为什么见它），双向 links
Pass 5: 质量检查 — 不空泛、不重复、不编造、links 有说明、source 可追溯
```

### 10.6 质控门禁

1. 建卡前必须先搜索旧卡（至少 3 轮搜索螺旋），判断是否已有相关卡。
2. content 不得空泛（禁止"本章讨论了 XXX 的重要性"类表述）。
3. links 必须在 content 中有对应的 `见 X（为什么见它）`。
4. 新建模式卡（gen 空间）必须引用 ≥ 3 张不同卡片，来自 ≥ 2 个不同主题分支或 namespace。
5. 不得编造材料中没有的信息。
6. source 必须清楚到能回到原始内容。
7. 产出后必须回头看：重新搜索确认无重复/冲突/遗漏，回到材料确认无重要遗漏。
8. 每批建卡后 Agent 自检 + 人抽检。

### 10.7 Agent 理解保障

Agent 必须充分理解本系统的前因后果、目的、精要才能工作。保障方式：

1. Agent 定义文件中写入系统目的、卡片设计精要、工作流程和反面示例。
2. Agent 启动时主动读取 `docs/design/000-system-overview.md`。
3. Agent 定义不是规则清单，而是包含"为什么这样设计"的理解性说明。

## 11. 实施阶段

### 阶段 1：存储层 + 工具脚本

- 建 SQLite schema
- 写 `tools.py`（search/get/create/update/link）
- 可选：迁移现有 JSON 卡片到 SQLite

### 阶段 2：Agent 定义

- 写 `.claude/agents/second-brain-digester.md`
- 包含系统理解（Why）+ schema 精要（What）+ 流程与质控（How）+ 反面示例
- 指向 `docs/design/000-system-overview.md` 作为深度参考

### 阶段 3：小样板验证

- 选一个小主题（如黏液瘤 / Spitz痣 / 某次事故复盘）
- 用 Agent 消化一段材料，产出 20-50 张卡
- 检查卡片质量：content 是否可用、links 是否有意义、type 是否准确、source 是否可追溯
- 迭代 Agent 定义

### 阶段 4：扩大

- 验证通过后逐步处理更多材料
- 每批抽检 10-20%
- 不合格则回滚该批并调整 Agent
- 系统持续演化：补充旧卡、重写、拆卡、合卡、新建模式、反思旧判断
